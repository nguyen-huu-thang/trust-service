## Overview

Trust Service exposes two groups of gRPC APIs:

| Group | Purpose |
|---|---|
| **External** | Service-to-service runtime: key distribution, cert rotation, trust bootstrap |
| **Internal** | Admin operations: register services, manage keys, certs, policies, shards |

Proto files are located in `src/main/proto/`:
- `external/` — public client APIs
- `internal/` — admin service definitions

## External APIs

### KeyDistributionService

```protobuf
service KeyDistributionService {
  // Fetch public keys to VERIFY JWT (cache and verify by kid)
  rpc GetPublicKeys(GetPublicKeysRequest) returns (GetPublicKeysResponse);

  // Fetch signing key to SIGN JWT (Identity Service only)
  rpc GetSigningKey(GetSigningKeyRequest) returns (GetSigningKeyResponse);
}

message GetPublicKeysRequest {
  string verifier_service_id = 1;
}

message GetPublicKeysResponse {
  repeated PublicKeyDto keys = 1;   // all unexpired public keys
}

message GetSigningKeyRequest {
  string signer_service_id = 1;
}

message GetSigningKeyResponse {
  repeated SigningKeyDto keys = 1;  // current + upcoming signing keys
}

message PublicKeyDto {
  string id = 1;                    // KSUID (used as kid in JWT header)
  string verifier_service_id = 2;
  string algorithm = 3;             // RSA or EC
  int32  key_size = 4;
  string public_key = 5;            // PEM format
  int64  activate_at = 6;           // Unix timestamp (ms)
  int64  expires_at = 7;
}

message SigningKeyDto {
  string id = 1;
  string signer_service_id = 2;
  string verifier_service_id = 3;
  string algorithm = 4;
  int32  key_size = 5;
  string private_key = 6;           // Identity Service only — encrypted in transit
  int64  activate_at = 7;
  int64  expires_at = 8;
}
```

### CertificateService

```protobuf
service CertificateService {
  // Rotate certificate (use when cert is approaching rotation threshold)
  rpc RotateCertificate(RotateCertificateRequest) returns (RotateCertificateResponse);
}

message RotateCertificateRequest {
  string token_id = 1;       // ID of the one-time refresh token
  string refresh_token = 2;  // one-time token value
  string new_public_key = 3; // PEM — new key pair generated by the service
  string service_id = 4;
}

message RotateCertificateResponse {
  CertificateDto certificate = 1;
  string next_refresh_token = 2;    // token for the next rotation
  string next_refresh_token_id = 3;
  int64  issued_at = 4;
  int64  expires_at = 5;
}

message CertificateDto {
  string id = 1;
  string service_id = 2;
  string public_cert = 3;   // PEM format — the full X.509 certificate
}
```

### TrustService

```protobuf
service TrustService {
  // Fetch Root CA certificate (bootstrap trust store, verify mTLS certs)
  rpc GetRootCertificate(GetRootCertificateRequest) returns (GetRootCertificateResponse);
}

message GetRootCertificateRequest {}

message GetRootCertificateResponse {
  string root_cert = 1;     // PEM format
  string fingerprint = 2;   // SHA-256 fingerprint (optional, for pinning)
  int64  expires_at = 3;
}
```

## Internal APIs (Admin)

Internal APIs are used by platform operators and automated tooling to manage Trust Service state. They are not called at runtime by other services.

```
KeyAdminService       ← manually generate / inspect keys
CertAdminService      ← bootstrap first cert for a new service
ServiceAdminService   ← register/deregister service IDs
PolicyAdminService    ← create / update key policies
ShardAdminService     ← register shards
```

Access is restricted to internal network only. mTLS + admin role required.

## API Usage Rules

### JWT Key Distribution

- Services must **cache** public keys locally — do not call `GetPublicKeys` on every request
- `GetPublicKeys` returns all unexpired public keys — cache all of them for backward compatibility
- `GetSigningKey` is for Identity Service only — no other service should call it
- Re-fetch public keys on a schedule (e.g., every 15 minutes), not on cache miss per request

### Certificate Management

```
New service bootstrap:
  1. Admin calls CertAdminService.BootstrapCert (internal API)
  2. Service receives cert + initial refresh token

Cert rotation (service-initiated):
  1. Service detects cert age (or is notified by lifecycle check)
  2. Service calls RotateCertificate with current mTLS + refresh token
  3. Service receives new cert + new refresh token
  4. Service updates its TLS configuration

Trust store bootstrap (first start):
  1. Service calls GetRootCertificate (unauthenticated)
  2. Service pins the Root CA in its trust store
```

### When to Call Trust Service

| Situation | Should call? |
|---|---|
| Verifying a JWT | No — use cached public keys |
| Signing a JWT | No (Identity Service only, and only for key refresh) |
| Service-to-service request | No — mTLS uses existing cert |
| First startup (trust store) | Yes — call `GetRootCertificate` |
| Cert approaching rotation | Yes — call `RotateCertificate` |
| Periodic key refresh | Yes — call `GetPublicKeys` on schedule |
| **Normal request path** | **Never** |
